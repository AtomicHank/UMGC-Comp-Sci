1. INTRODUCTION

This project implements a Java Swing GUI application for displaying and
calculating properties of two-dimensional and three-dimensional geometric
shapes. Users can select a shape from a dropdown menu, enter dimensional
parameters through text fields, and view the shape rendered in the application
window with calculated area or volume.

The project demonstrates advanced object-oriented programming concepts
including:
- Inheritance hierarchies with abstract and concrete classes
- Polymorphism through method overriding
- Factory design pattern for object creation
- Event-driven GUI programming with Swing
- Model-View-Controller (MVC) architecture
- Programmatic graphics rendering with Java 2D

Key Features:
- Nine geometric shapes: Circle, Square, Rectangle, Triangle, Sphere, Cube,
  Cone, Cylinder, Torus
- Interactive Swing GUI with event handlers
- Real-time shape rendering with 3D effects
- Automatic area/volume calculations
- Input validation and error handling
- Professional graphics with gradients and antialiasing

The project satisfies all required IS-A and HAS-A relationships as specified
in the assignment rubric.

7.6 Shape Calculations
----------------------
The application uses standard geometric formulas:

2D Shapes:
- Circle: Area = π × r²
- Square: Area = s²
- Rectangle: Area = width × height
- Triangle: Area = (base × height) / 2

3D Shapes:
- Sphere: Volume = (4/3) × π × r³
- Cube: Volume = s³
- Cone: Volume = (1/3) × π × r² × h
- Cylinder: Volume = π × r² × h
- Torus: Volume = 2 × π² × R × r² (R=major radius, r=minor radius)

All calculations use double precision floating-point arithmetic for accuracy.

8. DESIGN AND IMPLEMENTATION

This section documents the architectural decisions, design patterns, and
implementation details of the Shape Viewer application.

8.1 Class Hierarchy Design
---------------------------
The application uses a three-level inheritance hierarchy:

Level 1: Shape (Abstract Base Class)
- Properties: dimensions (int)
- Methods: getNumberOfDimensions(), getName() (abstract)
- Purpose: Common interface for all shapes

Level 2: TwoDimensionalShape and ThreeDimensionalShape (Abstract)
- TwoDimensionalShape:
  * Sets dimensions = 2
  * Defines abstract getArea() method
- ThreeDimensionalShape:
  * Sets dimensions = 3
  * Defines abstract getVolume() method
- Purpose: Separate 2D and 3D behavior

Level 3: Concrete Shape Classes (9 total)
- 2D: Circle, Square, Rectangle, Triangle
- 3D: Sphere, Cube, Cone, Cylinder, Torus
- Each implements required calculation methods
- Each stores relevant dimensions as private fields

8.2 IS-A Relationships (Inheritance)
-------------------------------------
The following inheritance relationships are demonstrated:

Circle IS-A TwoDimensionalShape IS-A Shape
Square IS-A TwoDimensionalShape IS-A Shape
Rectangle IS-A TwoDimensionalShape IS-A Shape
Triangle IS-A TwoDimensionalShape IS-A Shape
Sphere IS-A ThreeDimensionalShape IS-A Shape
Cube IS-A ThreeDimensionalShape IS-A Shape
Cone IS-A ThreeDimensionalShape IS-A Shape
Cylinder IS-A ThreeDimensionalShape IS-A Shape
Torus IS-A ThreeDimensionalShape IS-A Shape

Additionally:
ShapeGUI IS-A JFrame
ShapePanel IS-A JPanel

8.3 HAS-A Relationships (Composition)
--------------------------------------
Each shape HAS-A dimensional properties:

Circle HAS-A radius
Square HAS-A side
Rectangle HAS-A width and height
Triangle HAS-A base and height
Sphere HAS-A radius
Cube HAS-A side
Cone HAS-A radius and height
Cylinder HAS-A radius and height
Torus HAS-A major radius and minor radius

GUI components:
ShapeGUI HAS-A ShapePanel
ShapeGUI HAS-A JComboBox, JTextField, JButton, JLabel components
ShapePanel HAS-A Shape (for rendering)

8.4 Design Patterns Implemented
--------------------------------
8.4.1 Factory Pattern (ShapeFactory)
Purpose: Encapsulate object creation logic
Benefits:
- Client code doesn't need to know concrete shape classes
- Easy to add new shapes without modifying GUI code
- Centralizes parameter validation
- Follows Open/Closed Principle

Implementation:
- Static createShape() method takes shape name and parameters
- Returns Shape reference (polymorphism)
- Throws exceptions for invalid inputs

8.4.2 Model-View-Controller (MVC)
Purpose: Separate concerns for maintainability
Components:
- Model: Shape classes (business logic and calculations)
- View: ShapePanel (presentation and rendering)
- Controller: ShapeGUI (event handling and flow control)

Benefits:
- Each component has single responsibility
- Can modify rendering without affecting calculations
- Can change GUI layout without affecting shape logic

8.4.3 Template Method Pattern
Purpose: Define algorithm skeleton in base class
Implementation:
- getArea() defined in TwoDimensionalShape
- getVolume() defined in ThreeDimensionalShape
- Concrete classes provide specific implementations

Benefits:
- Guarantees all 2D shapes calculate area
- Guarantees all 3D shapes calculate volume
- Enforces consistent interface

8.5 Event-Driven Architecture
------------------------------
The GUI uses event-driven programming:

Events:
1. ItemEvent: Triggered when shape selection changes
   - Handler: itemStateChanged()
   - Action: Update parameter fields

2. ActionEvent: Triggered by button clicks
   - Handler: actionPerformed()
   - Sources: Display button, Clear button
   - Actions: Create/display shape or clear display

Listeners:
- ShapeGUI implements ItemListener for dropdown
- ShapeGUI implements ActionListener for buttons

Benefits:
- Responsive user interface
- Decoupled components
- Follows Swing best practices

10. LESSONS LEARNED

This section documents insights, challenges, and knowledge gained during the
development of the Shape Viewer GUI application.

10.1 Technical Lessons
----------------------
10.1.1 Inheritance Hierarchy Design
Lesson: Planning the hierarchy before coding saves significant refactoring.

The three-level hierarchy (Shape -> 2D/3D -> Concrete) was designed on paper
first. This prevented the need to restructure classes later. Starting with a
clear diagram helped identify what belongs in base classes versus concrete
classes.

Key Insight: Abstract classes should contain only truly common behavior.
Putting getArea() in Shape would have forced 3D shapes to implement it
unnecessarily.

Application: Always diagram inheritance hierarchies before implementation.

10.1.2 Factory Pattern Benefits
Lesson: Factory pattern significantly reduces coupling and improves
maintainability.

Initially considered creating shapes directly in the GUI (new Circle(), etc.).
The Factory pattern isolated shape creation, making the GUI independent of
specific shape constructors. Adding new shapes now requires only updating the
factory, not the GUI code.

Challenge: Learning when to use static vs instance factory methods.
Solution: Used static methods since no factory state needed to persist.

10.1.3 Event-Driven Programming
Lesson: Event handlers should be lightweight and delegate to helper methods.

Early versions had all logic in actionPerformed(). This became unreadable.
Refactoring to handleDisplayAction() and handleClearAction() improved clarity
and testability.

Best Practice: Event methods should route to specific handlers, not contain
business logic directly.

10.1.4 Java 2D Graphics Capabilities
Lesson: Java 2D provides powerful tools for creating professional graphics.

Discovering RadialGradientPaint transformed the sphere from flat to realistic.
Learning about BasicStroke, antialiasing, and color gradients elevated the
visual quality significantly.

Key Techniques Learned:
- Gradients for depth perception
- Antialiasing for smooth edges
- Isometric projection formulas
- Layering for 3D effects

Challenge: Creating torus shape without 3D graphics library.
Solution: Multiple ellipses with strategic layering and shading.

10.1.5 Input Validation Strategies
Lesson: Multi-layer validation provides better user experience and robustness.

Validating at GUI layer (type checking) AND domain layer (business rules)
catches different error categories. GUI layer provides immediate feedback,
domain layer ensures data integrity.

Pattern Learned:
1. Parse input (catch NumberFormatException)
2. Validate business rules (positive values)
3. Display user-friendly errors
4. Keep system in stable state

10.2 Design Lessons
-------------------
10.2.1 Separation of Concerns (MVC)
Lesson: Separating Model, View, and Controller makes each component easier to
understand and modify.

Shape classes know nothing about GUI. ShapePanel knows nothing about event
handling. ShapeGUI coordinates but doesn't do calculations. This separation
allowed rendering improvements without touching shape logic.

Benefit: Could swap Swing for JavaFX by replacing View and Controller only.

10.2.2 Encapsulation
Lesson: Private fields with getters protect object integrity.

All shape dimensions are private with public getters. No setters means shapes
are immutable after construction. This prevents invalid state (negative
dimensions) after object creation.

Pattern: Constructor validates, private fields preserve, getters expose.

10.2.3 Polymorphism in Action
Lesson: Polymorphism allows treating diverse objects uniformly.

ShapePanel's setShape(Shape s) accepts any shape type. The actual rendering
logic uses instanceof checks, but the interface is polymorphic. This allowed
the GUI to work with shapes it doesn't know about specifically.

Insight: Polymorphism at the interface level, specialization at the
implementation level.

10.2.4 Javadoc Documentation
Lesson: Writing documentation while coding is more effective than after.

Documenting each method immediately after writing it captured intent while
fresh in mind. Going back later to document would have been guesswork. Header
comments explaining formulas proved invaluable during testing.

Practice: Write class/method documentation immediately after writing the
signature.

10.3 Development Process Lessons
---------------------------------
10.3.1 Incremental Development
Lesson: Building in stages reduces complexity and debugging time.

Development order:
1. Shape hierarchy (no GUI)
2. Test shapes with simple text output
3. Basic GUI framework
4. Connect shapes to GUI
5. Add rendering
6. Enhance graphics
7. Polish error handling

Each stage was tested before proceeding. Finding a calculation error in stage 2
was easier than in stage 6.

10.3.2 Test-Driven Mindset
Lesson: Thinking about test cases while coding improves quality.

Considering "what if user enters -100?" while writing validation led to better
error handling. Thinking about edge cases (very small, very large values)
during rendering development prevented issues later.

Practice: Ask "how will I test this?" while writing each method.

10.3.3 Code Organization
Lesson: Consistent naming and structure makes navigation easier.

Using clear patterns:
- Classes: noun phrases (ShapeFactory)
- Methods: verb phrases (createShape, renderSphere)
- Event handlers: handle prefix (handleDisplayAction)

Made finding code intuitive even weeks later.

10.3.4 Version Control Awareness
Lesson: Frequent saves and backups prevent disaster.

Although Git wasn't required, manually saving versions (v1, v2, etc.) saved
time when an experimental change didn't work. Being able to revert to a known-
good state was invaluable.

Recommendation: Use version control for all future projects.

10.4 Specific Challenges and Solutions
---------------------------------------
10.4.1 Challenge: Torus Rendering Complexity
Problem: Torus (donut shape) is the most complex 3D shape to render in 2D.

Initial Attempt: Single ellipse with hole - looked flat.

Solution Evolution:
1. Added back half in darker color (partial depth)
2. Added front half in lighter color (better depth)
3. Added tube cross-sections (realistic shape)
4. Added vertical compression (isometric view)

Final result: Convincing 3D donut appearance.

Lesson: Complex visuals require iterative refinement. Start simple, enhance
gradually.

10.4.2 Challenge: Dynamic GUI Layout
Problem: Different shapes need different numbers of parameters.

Initial Approach: Show all fields always, disable unused.
Issue: Looked cluttered and confusing.

Solution: Show/hide fields based on shape selection using setVisible().

Lesson: Dynamic layouts are worth the extra code for better UX.

10.4.3 Challenge: Large Input Values
Problem: User enters radius=10000, shape overflows window.

Initial Response: Let Java handle it (result: ugly rendering).

Solution: Add Math.min() capping for each shape type.

Lesson: Defensive programming prevents poor user experience. Validate AND
constrain.

10.4.4 Challenge: Decimal Input Precision
Problem: Floating-point arithmetic can have rounding errors.

Consideration: Use BigDecimal for precise calculations?

Decision: double precision sufficient for this application (visual display,
not financial calculations).

Lesson: Choose appropriate precision level for the use case.

10.5 Professional Development Insights
---------------------------------------
10.5.1 Requirements Analysis
Lesson: Understanding requirements thoroughly before coding prevents rework.

Reading the rubric carefully revealed:
- GUI required (not just console)
- Event handlers required (specific Java interfaces)
- Both 2D and 3D shapes required
- Has-a and Is-a relationships required

Each requirement mapped to specific design decisions. Missing any would have
required major changes.

10.5.2 Code Readability
Lesson: Code is read more than written; optimize for readability.

Using descriptive variable names (majorR, minorR vs r1, r2) made formulas
self-documenting. Helper methods like render2DShape() made main paint method
scannable.

Principle: If you have to comment what code does, the code isn't clear enough.

10.5.3 Error Message Quality
Lesson: Error messages should guide users to solutions, not just state
problems.

Bad: "Invalid input"
Better: "Please enter valid numeric values for all parameters."
Best: "Please enter valid numeric values for all parameters." + highlighting
the problem field

Future improvement: Indicate which field has the problem.

10.5.4 User Experience Focus
Lesson: Technical correctness isn't enough; the user experience matters.

Correct but ugly renderings wouldn't meet the assignment goal. Gradients,
antialiasing, and careful color choices made the application visually
appealing. Clear status messages kept users informed.

Principle: Software is for users, not just compilers.
