1. Recursion tree for recursive bubble sort on an array of length 4

Given:

void bubbleSort(int array[]) {
  sort(array, 0);
}

void sort(int[] array, int i) {
  if (i < array.length - 1) {
     bubble(array, i, array.length - 1);
     sort(array, i + 1);
  }
}

void bubble(int[] array, int i, int j) {
  if (j <= i)
    return;
  if (array[j] < array[j - 1]) {
    int temp = array[j];
    array[j] = array[j - 1];
    array[j - 1] = temp;
  }
  bubble(array, i, j - 1);
}

For an array of length 4 in the worst case, such as [4, 3, 2, 1], the recursion tree is:

bubbleSort(array)
'- sort(array, 0)
   |- bubble(array, 0, 3)
   |  '- bubble(array, 0, 2)
   |     '- bubble(array, 0, 1)
   |        '- bubble(array, 0, 0)
   '- sort(array, 1)
      |- bubble(array, 1, 3)
      |  '- bubble(array, 1, 2)
      |     '- bubble(array, 1, 1)
      '- sort(array, 2)
         |- bubble(array, 2, 3)
         |  '- bubble(array, 2, 2)
         '- sort(array, 3)

Explanation:

- bubbleSort calls sort(array, 0).
- Each sort(array, i) performs one bubbling pass, then recursively calls sort(array, i + 1).
- Each bubble(array, i, j) call makes only one recursive call, so the bubble part is a chain rather than a branching subtree.

Best-case difference:

The recursion tree has the same shape in the best case. The method still makes the same recursive calls because there is no early stopping test. The only difference is that in the best case, the condition

array[j] < array[j - 1]

is always false, so no swaps occur.


2. Node count, time, height, and memory

Number of nodes:

For an array of length n:

- There is 1 bubbleSort activation.
- There are n sort activations: sort(0), sort(1), ..., sort(n - 1).
- For each i from 0 to n - 2, bubble(i, n - 1) makes n - i activations, including the base case bubble(i, i).

So the total number of bubble activations is:

sum from i = 0 to n - 2 of (n - i)
= n + (n - 1) + ... + 2
= n(n + 1) / 2 - 1

Therefore the total number of nodes is:

1 + n + (n(n + 1) / 2 - 1)
= n + n(n + 1) / 2
= (n^2 + 3n) / 2

Execution time:

The recursive version performs the same amount of comparison work as ordinary bubble sort, so its running time is:

Theta(n^2)

Height of the tree:

The longest root-to-leaf path is:

bubbleSort -> sort(0) -> bubble(0, n - 1) -> bubble(0, n - 2) -> ... -> bubble(0, 0)

That path contains:

- 1 bubbleSort activation
- 1 sort activation
- n bubble activations

So the height is:

- n + 2 nodes, or
- n + 1 edges

Thus the height is:

Theta(n)

Memory:

At any moment, the deepest call stack consists of one sort activation plus a chain of bubble activations, so the extra memory usage is:

Theta(n)


3. Java class SortedPriorityQueue

To make the implementation as efficient as possible with a sorted int array, the smallest value should be stored at the end of the used portion of the array. That allows remove() to return the smallest element in Theta(1) time with no shifting.

public class SortedPriorityQueue {
    private final int[] queue;
    private int size;

    public SortedPriorityQueue(int capacity) {
        queue = new int[capacity];
        size = 0;
    }

    public void add(int value) {
        if (size == queue.length) {
            throw new RuntimeException("Priority queue is full");
        }

        int i = size - 1;
        while (i >= 0 && queue[i] < value) {
            queue[i + 1] = queue[i];
            i--;
        }
        queue[i + 1] = value;
        size++;
    }

    public int remove() {
        if (size == 0) {
            throw new RuntimeException("Priority queue is empty");
        }

        size--;
        return queue[size];
    }

    public boolean isEmpty() {
        return size == 0;
    }
}

Why this is efficient:

- The array is kept in descending order.
- The smallest element is always at index size - 1.
- add may need to shift elements right, so its worst-case cost is Theta(n).
- remove simply decreases size and returns the last element, so its cost is Theta(1).


4. Efficiency of the sorting algorithm that uses SortedPriorityQueue

Algorithm:

void sort(int[] array)
{
  SortedPriorityQueue queue = new SortedPriorityQueue(100);
  for (int i = 0; i < array.length; i++)
    queue.add(array[i]);
  for (int i = 0; i < array.length; i++)
    array[i] = queue.remove();
}

Worst-case execution time:

Let n be the length of the input array.

Insertion phase:

Each add may need to shift all existing elements to keep the array sorted.

- 1st add costs Theta(1)
- 2nd add costs Theta(2)
- ...
- nth add costs Theta(n)

Total insertion cost:

1 + 2 + ... + n = n(n + 1) / 2 = Theta(n^2)

Removal phase:

Because remove() just returns the last occupied entry and decrements size, each removal is Theta(1).

There are n removals, so the total removal cost is:

Theta(n)

Overall time:

Theta(n^2) + Theta(n) = Theta(n^2)

Comparison with Java's PriorityQueue:

Java's PriorityQueue is heap-based.

- add is Theta(log n)
- remove is Theta(log n)

So using Java's PriorityQueue to sort n values takes:

n Theta(log n) + n Theta(log n) = Theta(n log n)

Conclusion:

The SortedPriorityQueue implementation above is less efficient than Java's PriorityQueue for sorting.

- This implementation: Theta(n^2)
- Java PriorityQueue implementation: Theta(n log n)
