I. INTRODUCTION

This study compares Quick Sort and Merge Sort using the Java benchmarking program created in Project 1. Both algorithms are efficient comparison-based sorting algorithms, but they differ in how they divide the data and in how sensitive they are to input order. Quick Sort partitions the array around a pivot and then recursively sorts the two partitions. Merge Sort recursively divides the array into halves and then merges the two sorted halves back together. In theory, Quick Sort often performs very well in practice because of strong cache behavior and low constant factors, while Merge Sort offers more predictable asymptotic behavior because its divide-and-conquer structure does not depend on pivot quality.

The benchmark program generated 12 evenly spaced input sizes from 100 through 1200. For each size, it generated 40 data sets and applied both algorithms to the exact same inputs. The program recorded a critical-operation count and elapsed time in nanoseconds for every run and verified that each sorted array was in nondecreasing order after the algorithm completed.


A. PSEUDOCODE

Quick Sort pseudocode:

QUICKSORT(A, low, high)
if low >= high
    return
pivotIndex = PARTITION(A, low, high)
QUICKSORT(A, low, pivotIndex - 1)
QUICKSORT(A, pivotIndex + 1, high)

PARTITION(A, low, high)
pivot = A[high]
i = low - 1
for j = low to high - 1
    compare A[j] to pivot
    if A[j] <= pivot
        i = i + 1
        swap A[i] and A[j]
swap A[i + 1] and A[high]
return i + 1

Merge Sort pseudocode:

MERGESORT(A, left, right)
if left >= right
    return
mid = floor((left + right) / 2)
MERGESORT(A, left, mid)
MERGESORT(A, mid + 1, right)
MERGE(A, left, mid, right)

MERGE(A, left, mid, right)
while both halves contain elements
    compare current left value and current right value
    copy smaller value into temporary array
copy remaining values from unfinished half
copy merged values back into original array


B. BIG-O

Quick Sort has an average-case time complexity of O(n log n), but in the worst case it degrades to O(n^2) if repeated poor pivots produce highly unbalanced partitions. The implementation used here always chooses the last element as the pivot, so it is theoretically vulnerable to that worst case even though the randomly generated data tends to avoid it.

Merge Sort has time complexity O(n log n) in the best, average, and worst cases. Its running time is more predictable because it always divides the array approximately in half and then performs a linear-time merge. Merge Sort does require additional auxiliary storage, so its space complexity is larger than Quick Sort’s in-place partitioning approach.

For both algorithms, the critical-operation count should grow on the order of n log n for randomly generated data. The time measurements should broadly follow the same trend, although JVM effects and operating-system scheduling noise can make actual timings less smooth than theoretical counts.


C. JVM WARM UP

Java benchmarking is sensitive to JVM warm-up because the first executions of a method may include class loading, bytecode interpretation, and just-in-time compilation. To reduce that distortion, the Project 1 benchmark performed 200 warm-up runs before any measurements were recorded. Both Quick Sort and Merge Sort were exercised during warm-up on small random arrays so that the JIT compiler had a chance to optimize the key sorting paths before the measured runs began. This does not eliminate all measurement noise, but it reduces the effect of startup costs on the reported times.


D. CRITICAL OPERATION

The critical operation counted for both algorithms was the element comparison that directly drives ordering decisions.

For Quick Sort, the counter was incremented every time an array value was compared to the pivot during partitioning.

For Merge Sort, the counter was incremented every time the current elements at the heads of the two subarrays were compared during merging.

This was a reasonable choice because comparisons are central to the behavior of both algorithms and provide a common basis for analysis. Counting comparisons also makes it easier to compare theoretical growth against observed behavior, since both algorithms are comparison sorts and their asymptotic efficiency is largely determined by how many ordering comparisons they perform.

B. PERFORMANCE COMPARISON

Quick Sort was generally faster in elapsed time across the measured input sizes even though its worst-case theoretical bound is weaker than Merge Sort’s. For every size in the final data set, the average execution time for Quick Sort was below the corresponding average for Merge Sort. This is consistent with the common practical observation that Quick Sort often performs better on in-memory arrays because it has low overhead and good locality of reference.

Merge Sort, however, showed a more uniform pattern in the critical counts. Its average comparison counts were lower than Quick Sort’s at every measured size, which is also expected. Even so, Merge Sort’s extra memory traffic and copying work during merging likely explains why it still took more nanoseconds on average in these runs.


C. CRITICAL OPERATION RESULTS AND TIME MEASUREMENTS

The critical-operation counts and the time measurements did not track one another perfectly. Merge Sort consistently used fewer counted comparisons than Quick Sort. For example, at size 1200, Merge Sort averaged 10,786.08 counted comparisons while Quick Sort averaged 13,625.58. Despite that, Merge Sort’s average execution time at size 1200 was 92,207.50 ns, compared with Quick Sort’s 69,975.00 ns.

This difference shows why it is useful to collect both types of data. The critical count captures one central theoretical cost, but it does not reflect all implementation overhead. Merge Sort copies data into a temporary array and back into the original array during every merge. Quick Sort performs more comparisons in this benchmark, but it works in place and may benefit from lower memory overhead. The time measurements therefore reveal constant-factor and memory-behavior costs that the critical-operation metric alone does not capture.


D. COEFFICIENT OF VARIANCE

The coefficient of variance for the critical-operation counts was very low for Merge Sort, ranging from 0.19% to 1.21%. That result shows Merge Sort is highly insensitive to the specific random data values used in this benchmark. Its divide-and-conquer structure is almost entirely determined by array size, so the count behavior is very stable across runs.

Quick Sort’s count variance was noticeably higher, ranging from 4.40% to 10.43%. That is still moderate, but it confirms that Quick Sort is more data-sensitive because the partition sizes depend on where the pivot falls relative to the input distribution.

The time coefficients of variance were much larger for both algorithms. Quick Sort’s time CV ranged from 1.97% to 78.85%, and Merge Sort’s time CV ranged from 2.61% to 148.44%. This is not surprising for nanosecond-level JVM timings. Timer granularity, garbage collection, background system activity, cache state, and JIT behavior all affect elapsed time more than they affect comparison counts. The large time CV at certain sizes therefore reflects measurement noise and runtime sensitivity rather than a contradiction of the theoretical analysis.


E. RESULTS vs BIG-O

The observed count trends were consistent with the expected O(n log n) growth for both algorithms on random input. Neither algorithm showed evidence of quadratic growth in the measured data. That is what should be expected for Merge Sort and also for Quick Sort on random data, where the average case dominates.

The time results were less smooth than the counts, but the overall pattern still supports the same broad conclusion. As input size increased, the average times generally increased as well, and Quick Sort usually retained an advantage in actual runtime. The experiment therefore supports the standard theoretical claim that both algorithms are efficient O(n log n) comparison sorts in typical cases, while also highlighting that asymptotic order alone does not determine real execution speed.


III. CONCLUSION

This study showed that Quick Sort and Merge Sort both behave consistently with O(n log n) growth on random inputs, but they differ in predictability and practical performance. Merge Sort produced lower and much more stable comparison counts, which reflects its deterministic structure and low data sensitivity. Quick Sort produced somewhat higher and more variable comparison counts, which is expected because partition quality depends on the input and pivot placement.

Despite that, Quick Sort was usually faster in measured nanoseconds. This indicates that implementation-level costs matter: in-place partitioning and lower auxiliary-memory overhead can outweigh the advantage of fewer comparisons. The coefficient-of-variance results also showed that critical-operation counts are more stable than raw timing measurements and therefore provide a cleaner view of theoretical growth. Overall, the benchmark confirms that both algorithms are efficient, but it also shows why real performance analysis should consider theory, counted operations, and actual run time together rather than relying on only one measure.
