I. INTRODUCTION

This study compares Quick Sort and Merge Sort using the Java benchmark from Project 1. Both are comparison-based divide-and-conquer sorting algorithms, but they differ in how they organize work. Quick Sort partitions the array around a pivot and then recursively sorts the two partitions. Merge Sort recursively divides the array into halves and then merges sorted halves back together. In practice, Quick Sort often has strong cache behavior and low constant factors, while Merge Sort provides more predictable growth because its recursive structure does not depend on pivot quality.

The benchmark generated 12 evenly spaced input sizes from 100 through 1200. For each size, it generated 40 random data sets and used the exact same input data for both algorithms. The program recorded one critical-operation count and one elapsed time in nanoseconds for every run, and it verified after each run that the output array was sorted correctly.


A. PSEUDOCODE

Quick Sort pseudocode:

QUICKSORT(A, low, high)
if low >= high
    return
pivotIndex = PARTITION(A, low, high)
QUICKSORT(A, low, pivotIndex - 1)
QUICKSORT(A, pivotIndex + 1, high)

PARTITION(A, low, high)
pivot = A[high]
i = low - 1
for j = low to high - 1
    compare A[j] to pivot
    if A[j] <= pivot
        i = i + 1
        swap A[i] and A[j]
swap A[i + 1] and A[high]
return i + 1

Merge Sort pseudocode:

MERGESORT(A, left, right)
if left >= right
    return
mid = floor((left + right) / 2)
MERGESORT(A, left, mid)
MERGESORT(A, mid + 1, right)
MERGE(A, left, mid, right)

MERGE(A, left, mid, right)
while both halves contain elements
    compare current left value and current right value
    copy smaller value into temporary array
copy remaining values from unfinished half
copy merged values back into original array


B. BIG-THETA

Merge Sort runs in Theta(n log n) time in the best, average, and worst cases. Its recursive division pattern is fixed by the input size, so the amount of comparison work grows predictably. Its extra space usage is Theta(n) because it needs auxiliary storage during merging.

Quick Sort has expected Theta(n log n) performance on random input, which is the behavior this benchmark is intended to observe. However, because the implementation chooses the last element as the pivot, its worst case is Theta(n^2) if repeated bad pivots create highly unbalanced partitions. Its auxiliary space usage is lower than Merge Sort's because it sorts in place except for recursion stack overhead.

For the random data used here, both algorithms should show growth consistent with Theta(n log n) in the critical-operation counts. The measured times should generally follow the same trend, although timing data is usually noisier than operation counts because of JVM and system effects.


C. JVM WARM UP

Java benchmarking is sensitive to JVM warm-up because early method calls may include class loading, bytecode interpretation, and just-in-time compilation overhead. To reduce that distortion, the Project 1 benchmark performed 200 warm-up runs before recording any measured data. Both Quick Sort and Merge Sort were exercised during warm-up on small random arrays so that the JVM had an opportunity to optimize the main sorting paths before the timed runs began.

This warm-up process does not remove all noise, but it makes the recorded time measurements more representative of steady-state execution.


D. CRITICAL OPERATION

The critical operation counted for both algorithms was the ordering comparison that directly determines element placement.

For Quick Sort, the counter was incremented every time an array value was compared to the pivot during partitioning.

For Merge Sort, the counter was incremented every time the current front elements of the left and right subarrays were compared during merging.

This was a good choice because both algorithms are comparison sorts, and comparison count is closely tied to their asymptotic behavior. Counting the same general kind of operation for both algorithms also makes the comparison fairer and easier to interpret.

B. PERFORMANCE COMPARISON

Quick Sort was faster in average elapsed time at every tested input size in the current data set, even though Merge Sort used fewer counted comparisons. For example, at size 1000, Quick Sort averaged 54,465.00 ns while Merge Sort averaged 77,400.00 ns. At size 1200, Quick Sort averaged 66,697.50 ns while Merge Sort averaged 94,350.00 ns.

Merge Sort was more efficient in terms of the chosen critical operation. At every size, its average comparison count was lower than Quick Sort's. This matches standard expectations: Merge Sort's comparison structure is highly regular, while Quick Sort's partitioning behavior depends more on how the pivot divides the input.

Overall, the benchmark shows a familiar practical tradeoff. Merge Sort performs fewer counted comparisons, but Quick Sort still finishes sooner because its in-place partitioning has lower overhead than Merge Sort's repeated copying into and out of temporary storage.


C. CRITICAL OPERATION RESULTS AND TIME MEASUREMENTS

The comparison counts and time measurements tell related but not identical stories. Merge Sort consistently used fewer counted comparisons than Quick Sort, but it still took longer in elapsed nanoseconds. At size 1200, Merge Sort averaged 10,786.08 counted comparisons while Quick Sort averaged 13,625.58. Even so, Merge Sort averaged 94,350.00 ns, while Quick Sort averaged 66,697.50 ns.

This difference shows why benchmarking should collect both a theoretical cost metric and a real time measurement. Comparison count tracks one central aspect of algorithmic growth, but it does not capture all of the work the implementation performs. Merge Sort repeatedly copies values into a scratch array and back into the original array. Quick Sort performs more comparisons here, but it stays in place and avoids that additional memory traffic.

The time data therefore exposes constant-factor costs and memory-behavior costs that do not appear directly in the comparison counts.


D. COEFFICIENT OF VARIANCE

The coefficient of variance for Merge Sort's comparison counts was extremely low, ranging from 0.19% to 1.21%. That result shows Merge Sort was highly insensitive to the particular random values in the input data. Its recursive structure depends mainly on array size, so its comparison behavior remains very stable across runs.

Quick Sort's count CV ranged from 4.40% to 10.43%, which is still moderate but clearly higher. That confirms that Quick Sort is more data-sensitive because the sizes of the partitions depend on where the pivot falls relative to the data.

The time CV values were much larger for both algorithms. Quick Sort's time CV ranged from 1.82% to 49.13%, while Merge Sort's time CV ranged from 2.29% to 116.93%. Those larger time CVs reflect JVM scheduling noise, cache effects, garbage collection, timer granularity, and other runtime factors that affect nanosecond measurements much more than they affect comparison counts.

The coefficient-of-variance results therefore support two conclusions:

1. Comparison counts are the cleaner indicator of each algorithm's structural behavior.
2. Merge Sort is much less sensitive to input data than Quick Sort when the chosen metric is comparison count.


E. RESULTS vs BIG-THETA

The observed comparison counts are consistent with the expected Theta(n log n) growth for both algorithms on random input. Neither algorithm showed comparison-count growth suggesting Theta(n^2) behavior in this benchmark. That is exactly what should be expected for Merge Sort and also for Quick Sort under random data where the expected case dominates.

The time measurements were less smooth than the counts, but their overall trend still supported the same high-level conclusion. Average execution times generally increased as n increased, and Quick Sort maintained a practical runtime advantage throughout the tested range.

The experiment therefore supports the theoretical analysis well. Merge Sort behaved like the more predictable algorithm, and Quick Sort behaved like the practically faster one on these inputs even though it has the weaker worst-case bound.


III. CONCLUSION

This benchmark showed that both Quick Sort and Merge Sort behaved consistently with Theta(n log n) growth on random input sizes from 100 through 1200. Merge Sort produced lower and much more stable comparison counts, which reflects its deterministic recursive structure and low data sensitivity. Quick Sort produced somewhat higher and more variable comparison counts, which is expected because its partition quality depends on the input and pivot placement.

Despite that, Quick Sort was faster in average elapsed time at every measured input size in the current data set. That result shows that implementation-level details matter: fewer comparisons do not automatically mean lower run time. In this benchmark, Quick Sort's in-place behavior offset the fact that it performed more comparisons, while Merge Sort's extra memory movement increased its time cost.

Overall, the study confirms that both algorithms are efficient comparison sorts, but it also shows why algorithm analysis is strongest when asymptotic behavior, counted operations, data sensitivity, and actual run time are all considered together.
